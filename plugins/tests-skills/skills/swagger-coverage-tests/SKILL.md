# Генерация API тестов на основе отчёта swagger-coverage

Навык анализирует отчёт о покрытии API тестами и генерирует недостающие тесты, используя правила из навыка `api-tests-swagger`.

## Обязательные шаги

### Шаг 1. Запросить входные данные

Если пользователь не указал пути, спроси:

1. **Путь к отчёту покрытия** — JSON файл `swagger-coverage-results.json` (или аналогичный). Обычно лежит в `build/coverage-output/` или в корне проекта.
2. **Путь к Swagger спецификации** — файл `catalog4.swagger.json` (или аналогичный). Обычно лежит в `src/test/resources/`.

Если пользователь указал только один файл — спроси про второй.

### Шаг 2. Прочитать и проанализировать отчёт покрытия

Прочитай JSON файл покрытия. Он имеет следующую структуру:

```json
{
  "api": "Название API",
  "generated_at": "2026-05-07 13:55:41",
  "summary": {
    "total_operations": 89,
    "fully_covered": 5,
    "partially_covered": 20,
    "not_covered": 128,
    "coverage_percent": 13.0
  },
  "paths": {
    "/endpoint/path": {
      "GET": {
        "state": "EMPTY | PARTY | FULL",
        "coverage": "0/16",
        "deprecated": false,
        "requirements": {
          "status_codes": [{ "code": "200", "covered": false, "description": "OK" }],
          "parameters": [{ "in": "query", "covered": false, "name": "param_name", "description": "Описание" }],
          "body": [],
          "properties": []
        }
      }
    }
  }
}
```

**Состояния операций (`state`):**
- `EMPTY` — операция полностью не покрыта (нет ни одного вызова)
- `PARTY` — частично покрыта (вызовы есть, но не все параметры/статусы)
- `FULL` — полностью покрыта

**Приоритет генерации:** сначала `EMPTY` (нет тестов вообще), затем `PARTY` (нужно дополнить).

### Шаг 3. Вывести сводку покрытия

Покажи пользователю:
- Общий процент покрытия
- Количество непокрытых / частично покрытых / полностью покрытых операций
- Список непокрытых операций (path + method + coverage ratio)

Спроси, какие операции покрыть тестами. Предложи варианты:
- Все `EMPTY` операции
- Конкретные эндпоинты (пользователь выбирает)
- Все `EMPTY` + `PARTY`

### Шаг 4. Прочитать Swagger спецификацию

Прочитай Swagger/OpenAPI спецификацию, чтобы получить:
- Параметры эндпоинта (типы, обязательность, значения по умолчанию)
- Модели запросов и ответов
- Теги (для группировки тестов)
- Описания операций

Это нужно для корректной генерации тестового кода — имена API методов, модели, параметры.

### Шаг 5. Найти существующие тесты

Найди в проекте существующие тестовые классы для анализируемых эндпоинтов. Поищи по:
- Именам файлов в `src/test/java/api/http/`
- Содержимому — вызовам API методов

Это нужно чтобы:
- Добавить новые тесты в существующий класс (если он есть)
- Не дублировать уже существующие тесты
- Понять паттерны мокирования и настройки, используемые в проекте

### Шаг 6. Сформулировать требования и передать в api-tests-swagger

Для каждой выбранной операции сформулируй список непокрытых требований — все записи с `covered: false` из `requirements`. Структура отчёта подробно описана в **[references/coverage-requirements.md](references/coverage-requirements.md)**.

Для каждого требования определи:
- Тип требования (status_code, parameter, body, property)
- Что нужно сделать для покрытия (какой статус вернуть, какой параметр передать, какое поле проверить)

Затем вызови навык `api-tests-swagger` и передай ему сформулированные требования. Этот навык содержит все технические правила генерации тестового кода: аннотации, executeAs/execute, проверки, именование, форматирование.

**Правила:**
- Тесты должны быть атомарны — каждый тест направлен на покрытие конкретных требований
- Если существующий тестовый класс найден — добавь метод в него. Если нет — создай новый класс.

### Шаг 7. Представить результат

Покажи сгенерированные тесты пользователю. Для каждого теста укажи:
- Какой эндпоинт покрывается
- Какое состояние покрытия было (EMPTY/PARTY)
- Что именно проверяется

## Важные правила

- **Всегда** вызывай навык `api-tests-swagger` перед генерацией кода — он задаёт стандарты написания тестов в проекте
- Не генерируй тесты для `deprecated` операций (проверяй поле `deprecated: true`)
- Если Swagger спецификация не содержит описания для эндпоинта — уточни у пользователя
- Проверяй, что сгенерированные API методы реально существуют в проекте (сравнивай с OpenAPI генератором)
- Группируй тесты по эндпоинтам (path), а не по отдельным параметрам
- **Один API вызов на тест** — никогда не вызывай один и тот же API метод дважды в одном тесте.
- `@DisplayName` класса — название метода/эндпоинта, например `@DisplayName("Healthz")`. Передавай этот контекст в навык `api-tests-swagger`.
- Все технические детали реализации (execute/executeAs, паттерны вызова, проверки) — в навыке `api-tests-swagger`. Этот навык только формулирует требования.
