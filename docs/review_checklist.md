# Чек-лист для проверки формулировок в ревью-комментариях

**Назначение:** Проверка корректности и однозначности формулировок в комментариях ревью тест-кейсов перед публикацией.

---

## 📋 Общие принципы

### ✅ Правильные формулировки

| Тип действия | ❌ НЕПРАВИЛЬНО | ✅ ПРАВИЛЬНО | Почему |
|--------------|---------------|-------------|--------|
| **Перемещение данных** | "Удалить из описания информацию об участниках" | "**Перенести** информацию об участниках из описания в специальное поле «Участники»" | Информация не теряется, а перемещается в правильное место |
| **Вынос ссылок** | "Убрать ссылки из описания" | "**Вынести** ссылки из описания в поле «Ссылки»" | Чётко указано место назначения |
| **Участники** | "Добавить аналитика" | "Добавить участника: username «a.stukalin», role_id 3 (Аналитик)" | Конкретные идентификаторы вместо неопределённости |
| **JSON/XML** | "Удалить JSON из текста" | "**Вынести** JSON-структуру во вложение к шагу с названием «Пример ответа»" | Понятно, что данные сохраняются |
| **Шаги** | "Изменить шаг 1" | "Для шага 1: изменить ожидаемый результат на «В ответе возвращается объект с ошибкой 500»" | Точное описание изменения |

## 🔍 Детальный чек-лист

### 1. Проверка формулировок по секциям

#### 1.1. Имя тест-кейса

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| Название конкретное | "Исправить название" | "Название: «Обновление достижения с неверным авторизационным токеном»" |
| Нет технических дубликатов | "Название (клон)(клон) 500" | "Название: «Обновление достижения с ошибкой»" |
| Рекомендуемая длина | "Слишком длинное название" | "Название превышает 150 символов, сократить до: «...»" |

#### 1.2. Описание

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| **Перемещение участников** | "Удалить информацию об участниках" | "**Перенести** информацию об участниках из описания в специальное поле «Участники»" |
| **Вынос ссылок** | "Убрать ссылки" | "**Вынести** ссылки [https://...] в поле «Ссылки» (или «Связанные задачи» для Jira/Tracker)" |
| **Сохранение контента** | "Удалить текст" | "**Переместить** текст в подходящее поле/секцию" |

#### 1.3. Предусловие

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| **Вынос технических деталей** | "Удалить cURL и body" | "**Вынести** cURL во вложение к шагу 2 с названием «cURL запроса»" |
| **Вынос JSON/XML** | "Убрать JSON из предусловия" | "**Вынести** JSON-структуру ответа во вложение к шагу 2.1 с названием «Пример ответа 500»" |
| **Сохранение сути** | "Удалить всё" | "Оставить краткое описание начального состояния: «Необходим существующий ID достижения»" |

#### 1.4. Ссылки

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| **Тип ссылки** | "Добавить ссылку" | "Добавить ссылку https://tracker.yandex.ru/FULLSTACK-27085 в поле «Связанные задачи» (интеграция с Yandex Tracker)" |
| **Название** | "Вынести ссылку" | "**Вынести** ссылку https://... в поле «Ссылки» с названием «Требования»" |

#### 1.5. Участники

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| **Конкретные идентификаторы** | "Добавить аналитика: Иванов" | "Добавить участника: username «a.stukalin», role_id 3 (Аналитик)" |
| **Формат данных** | "Добавить testers" | "Добавить участницу: username «AlenaKanchurova», role_id 7 (QA)" |

#### 1.6. Шаги

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| **Удаление** | "Удалить шаг 1" | "Удалить пустой шаг 1 без описания действия" |
| **Перемещение** | "Переместить вложение" | "**Переместить** вложение с примером ответа из подшага 1.1 в ожидаемый результат шага 1" |
| **Вынос во вложения** | "Убрать JSON" | "**Вынести** JSON-структуру {\"message\": \"...\"} во вложение к шагу 2 с названием «Пример ответа»" |
| **Изменение expected_result** | "Изменить результат" | "**Изменить** ожидаемый результат на краткое описание: «В ответе возвращается объект с ошибкой 500 Internal Server Error»" |

#### 1.7. Ожидаемый результат

| ✅ Проверка | ❌ Неправильно | ✅ Правильно |
|-------------|---------------|-------------|
| **Конкретная формулировка** | "Исправить результат" | "**Изменить** ожидаемый результат на: «Статус код 200 OK, тело содержит объект достижения»" |
| **Вынос во вложения** | "Удалить JSON из результата" | "**Вынести** JSON-структуру во вложение с названием «Пример успешного ответа»" |

### 2. Проверка структуры комментария

#### ✅ Обязательные элементы

- [ ] **Нумерация секций:** 1, 2, 3, 4, 5, 6, 7
- [ ] **Названия секций:** Имя тест-кейса, Описание, Предусловие, Ссылки, Участники, Шаги, Ожидаемый результат
- [ ] **Разделение "Проблемы" и "Предлагаемые правки":**
  - Сначала описание проблем (что не так)
  - Затем конкретные правки (как исправить)

#### ✅ Формат "Предлагаемых правок"

```markdown
Предлагаемые правки:
1. Имя тест-кейса
   - Название: «Конкретное название»

2. Описание
   - Перенести информацию об участниках в поле «Участники»
   - Вынести ссылки в поле «Ссылки»

3. Предусловие
   - Вынести cURL во вложение к шагу 2 с названием «cURL запроса»
   - Оставить краткое описание: «Необходим существующий ID»

5. Участники
   - Добавить участника: username «a.stukalin», role_id 3 (Аналитик)

6. Шаги
   - Удалить пустой шаг 1
   - Шаг 2:
     - Изменить ожидаемый результат на: «Краткое описание без JSON»
     - Вынести JSON во вложение с названием «Пример ответа»
```

### 3. Проверка на опасные формулировки

#### ❌ Запрещённые слова без контекста

| Слово | Опасность | ✅ Как исправить |
|-------|-----------|-----------------|
| "Удалить" | Потеря данных | "**Перенести** в [место]" или "**Вынести** в [поле/вложение]" |
| "Убрать" | Неоднозначно | "**Переместить** в [конкретное место]" |
| "Изменить" | Неясно что | "**Изменить [конкретное поле] на [конкретное значение]**" |
| "Исправить" | Неочевидно как | "**Заменить на: [конкретный текст]**" |
| "Добавить" (без деталей) | Недостаточно информации | "**Добавить [что] с названием [название]**" |

#### ✅ Безопасные формулировки с контекстом

| Правильная формулировка | Пример |
|-------------------------|--------|
| "**Перенести** [что] **из** [откуда] **в** [куда]" | "Перенести ссылки из описания в поле «Ссылки»" |
| "**Вынести** [что] **в** [куда] **с названием** [название]" | "Вынести JSON во вложение с названием «Пример ответа»" |
| "**Заменить** [что] **на** [новое значение]" | "Заменить ожидаемый результат на: «Статус код 200»" |
| "**Добавить** [что] **с параметрами** [детали]" | "Добавить участника: username «...», role_id 3" |

---

### 4. Проверка автоматизируемости

#### ✅ Критерий: может ли AI выполнить это действие?

| Формулировка | Автоматизируемо? | Почему |
|--------------|------------------|--------|
| "Перенести ссылки из описания в поле «Ссылки»" | ✅ Да | Есть MCP tools, чёткое действие |
| "Добавить участника: username «a.stukalin», role_id 3" | ✅ Да | Есть MCP tool, конкретные данные |
| "Улучшить описание" | ❌ Нет | Субъективная оценка |
| "Сделать название лучше" | ❌ Нет | Нет критериев "лучше" |
| "Изменить ожидаемый результат на: «В ответе возвращается объект с ошибкой 500»" | ✅ Да | Конкретный текст для замены |

---

### 5. Финальная проверка перед публикацией

#### ✅ Чек-лист готовности

- [ ] **Все формулировки конкретны** (нет "улучшить", "исправить" без деталей)
- [ ] **Нет слов "удалить"** без указания переноса данных (кроме действительно пустых элементов)
- [ ] **Все перемещения данных** имеют явное указание места назначения
- [ ] **Все добавления** имеют конкретные параметры (username, role_id, названия вложений)
- [ ] **Структура соответствует шаблону** (нумерация 1-7, разделение проблем/правок)
- [ ] **JSON/XML выносятся во вложения**, не удаляются
- [ ] **Ссылки выносятся в поля**, не удаляются из контекста
- [ ] **Участники переносятся из описания** в специальное поле, не удаляются
- [ ] **Технические детали** (cURL, body, headers) выносятся во вложения
- [ ] **Каждое действие может быть выполнено AI** без уточнений

## 📚 Примеры: Плохо → Хорошо

### Пример 1: Участники

❌ **Плохо:**
```markdown
2. Описание
   - Удалить информацию об участниках (Стукалин Антон, Канчурова Алена)
```

✅ **Хорошо:**
```markdown
2. Описание
   - Перенести информацию об участниках из описания в специальное поле «Участники»

5. Участники
   - Добавить участника: username «a.stukalin», role_id 3 (Аналитик)
   - Добавить участницу: username «AlenaKanchurova», role_id 7 (QA)
```

---

### Пример 2: JSON в шагах

❌ **Плохо:**
```markdown
6. Шаги
   - Шаг 2: убрать JSON из ожидаемого результата
```

✅ **Хорошо:**
```markdown
6. Шаги
   - Шаг 2:
     - Вынести JSON-структуру {"message": "Missing Authorization header"} во вложение с названием «Пример ответа 500»
     - Изменить ожидаемый результат на краткое описание: «В ответе возвращается объект с ошибкой 500 Internal Server Error»
```

---

### Пример 3: Ссылки

❌ **Плохо:**
```markdown
2. Описание
   - Убрать ссылки из текста
```

✅ **Хорошо:**
```markdown
2. Описание
   - Вынести ссылку https://tracker.yandex.ru/FULLSTACK-27085 в поле «Связанные задачи» (интеграция с Yandex Tracker)
   - Вынести ссылку https://wiki.example.com/docs/api в поле «Ссылки» с названием «Документация API»
```

---

### Пример 4: Предусловие

❌ **Плохо:**
```markdown
3. Предусловие
   - Удалить технические детали (cURL, body, ответ)
```

✅ **Хорошо:**
```markdown
3. Предусловие
   - Вынести cURL во вложение к шагу 2 с названием «cURL запроса»
   - Вынести JSON-body во вложение к шагу 2 с названием «Body запроса»
   - Вынести ответ JSON во вложение к шагу 2.1 (expected_result) с названием «Пример ответа 500»
   - Оставить краткое описание начального состояния: «Необходим существующий ID достижения»
```

## 🔧 Интеграция с промптом comment_t.md

После внедрения этого чек-листа рекомендуется добавить в `comment_t.md`:

```markdown
## Шаг 1.5. Проверка формулировок (опционально)

**КРИТИЧЕСКИ ВАЖНО:** Если комментарий содержит формулировки с "удалить", "убрать" без указания переноса данных:
1. Замените их на "перенести", "вынести" с указанием места назначения
2. Убедитесь, что все данные сохраняются (не удаляются безвозвратно)
3. Проверьте, что все добавления имеют конкретные параметры

Используйте чек-лист из `.claude/skills/review_comment_checklist.md` для проверки.
```

## 📊 Метрики качества

Отслеживайте количество комментариев с некорректными формулировками:

| Метрика | Цель | Как измерять |
|---------|------|--------------|
| % комментариев с "удалить" без контекста | < 5% | Поиск по тексту |
| % автоматизируемых правок | > 80% | Анализ секции "Предлагаемые правки" |
| % правок с конкретными параметрами | > 90% | Наличие username, role_id, названий |
| Время на уточнение формулировок | < 10% | Обратная связь от AI |
